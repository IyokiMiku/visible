## 阶段四：抽取共享模块并去硬编码

> 原则：把三套工具里"相同功能"的代码抽到 `backend/shared/`，全部改为接收 `ProjectContext`/`settings`，删除绝对路径、写死省份/考类、明文密钥。

### 步骤 4.1 — `shared/xueke_api/`（学科网拉题，三类型共用）
- 移植：`query_questions.py`、`api_pull_core.py`、`kpoint_resolver.py`（来自 考纲百套卷 `学科网API拉题移植版`）。
- 改造：`BASE`→`ctx` 项目树；`cookie/app_key/sign`→`config.get_xueke_config()`；修复失效 import。
- 暴露：`pull_for_plan(ctx, plan, needed) -> PulledResult`。

**验证**：给定 kpointId + 题型/难度，能拉到题（或在无 cookie 时给出明确错误）。

### 步骤 4.2 — `shared/docx/`（组卷底层）
- 移植：`docx_utils1.py`、`docx_generation.py`。
- 改造：命名/标题/页眉页脚/首部由 `ctx` + 类型模板决定；移除写死"考纲百套卷"后缀。

**验证**：传入 `ctx`（三种类型）分别产出正确文件名与三行标题。

### 步骤 4.3 — `shared/qc/`（质检）
- 移植：`质检/rules.py`、`quality.py`、`report.py`。
- 改造：接收 `ctx`；报告写入项目树 `质检报告/`、`错误报告.md`。

**验证**：对一份样卷跑质检，产出报告与 `pending_review` 标记。

### 步骤 4.4 — `shared/ai/`（LLM/补题/映射表）
- 移植：`config_io.call_api`、`prompts.py`、`generate_mapping.py`。
- 改造：key/base_url/model 从 settings；省份/考类参数化；`ai_fill(ctx, plan, shortfall)` 统一入口。

**验证**：`call_api` 连通；`generate_mapping(ctx)` 产出 `{映射表}.xlsx`。

### 步骤 4.5 — `shared/ocr/`（教材目录扫描 + 预留视觉）
- 移植：`textbook_toc_scanner.py`（本地 OCR）。
- 新增占位：`vision_ocr(images, cfg)`（读 `settings.vision`，未配置则抛"未启用"）。

**验证**：本地 OCR 跑通；`plan_source="ai_vision"` 时返回明确"未启用"提示。

## 阶段五：流程步骤实现（`engine/steps/`）

每个 step 一个文件，输入 `ctx`（与 `paper_no`），调用共享模块/工具，写产物 + 日志。

| 步骤文件 | 职责 |
|----------|------|
| `steps/kpoint_count.py` | 知识点数量文档（考纲/双析=学科网统计；一课一练=教材OCR扫描） |
| `steps/planning.py` | 规划表（本地OCR生成 / 上传xlsx 校验导入） |
| `steps/mapping.py` | 映射表（三类型） |
| `steps/mesh.py` | 细目表（仅考纲） |
| `steps/naming.py` | 确认考试名称/类型 → 写 `exam_type_name`、回写规划表 |
| `steps/pull.py` | 拉题（×倍率）+ 不足 AI 补题；低信度/不足→ `review` |
| `steps/assemble.py` | 组卷（考纲/一课一练 ×1） |
| `steps/split.py` | 奇偶分卷（仅双析；偶→教师、奇→学生，4 份） |
| `steps/qc.py` | 逐卷质检 + 报告 + 状态 |

**验证**：每个 step 可单测（mock ctx + 小样本）产出对应文件。

### 步骤 5.x — `engine/review.py`（人工确认队列）
- `enqueue(ctx, run_id, node, type, paper_no, confidence, payload)`；`confirm/return`。
- 类型：`AI_MATCH / AI_GENERATE / RULE_CONFLICT / QC_FAIL`。

**验证**：低于信度阈值时生成 `review_items(pending)` 且流程暂停。

### 步骤 5.y — `engine/runner.py`（编排）
- 按类型 `flow_nodes` 顺序调度 steps；写 `runs/flow_logs`；WebSocket 推送；遇 review 暂停；支持 `resume/rerun(node)`。

**验证**：一个项目能从头跑到 `done`，中途命中一次 review 暂停并可恢复。

## 阶段六：三类型驱动接入（`engine/drivers/`）

### 步骤 6.1 — `KaogangDriver`
- 包装考纲百套卷各阶段（规划表/映射表/细目表/拉题/组卷/质检）→ steps。
- `flow_nodes = [读取资料, 解析考纲, 生成规划, 知识点匹配, 细目表, 拉题与补题, 质检导出]`。

**验证**：跑通 1 卷（考点训练卷）出解析版+原卷版。

### 步骤 6.2 — `YikeyilianDriver`
- 复用教材目录扫描 + 8 列规划表；**新增 `gen_mapping`**（教材主题→kpointId，用 `shared/ai/generate_mapping` + `kpoint_resolver`）。
- 出题改走 `shared/xueke_api`；AI 补题用其 `build_generation_prompt`（移到 `shared/ai` 作为该类型的 fill 实现）。
- `flow_nodes = [读取资料, 解析目录, 生成规划, 知识点匹配, 拉题与补题, 质检导出]`。

**验证**：`第N练 …（解析版）` 文件名正确；卷号范围生效。

### 步骤 6.3 — `ShuangxiDriver`
- 规划表/映射表/2×拉题借共享（考纲方式）；`assemble` 调 `考点双析卷工具包/scripts/process_exam.py` 奇偶拆分。
- **翻转**奇偶映射为"偶→教师讲解卷、奇→学生练习卷"。
- `flow_nodes = [读取资料, 解析考纲, 生成规划, 知识点匹配, 拉题与补题, 奇偶分卷, 质检导出]`。

**验证**：seq=1 产出 第1卷(教师/偶)、第2卷(学生/奇)，各解析版+原卷版共 4 份。
