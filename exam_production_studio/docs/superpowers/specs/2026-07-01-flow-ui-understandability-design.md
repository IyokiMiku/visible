# 流程执行页可理解性改造设计

日期：2026-07-01

## 背景

`To-do.md` 中「流程执行界面」提出：待确认界面需要说人话、工作时要显示当前状态、工作中不允许点击继续、状态改成中文、考虑上方进度展示。当前 `frontend/src/views/Flow.vue` 已有流程步骤、日志、进度、按钮和待确认入口，但多个状态直接展示英文/内部节点名，主要依赖日志让用户判断系统是否仍在工作。

## 目标

本阶段先做低风险的前端可理解性改造，让用户能在流程执行页直接判断：当前是否在工作、正在做什么、下一步该怎么操作、哪些按钮暂时不能点。

## 非目标

- 不重做完整流程引擎。
- 不改变后端 API 或 WebSocket 事件格式。
- 不实现跨页面全局进度条；资源导入页与项目切换进度保留到后续统一设计。
- 不修改流程节点的真实执行逻辑。

## 推荐方案

在 `frontend/src/views/Flow.vue` 内增加一个轻量展示层：

1. 建立状态、节点、卷状态的中文映射。
2. 增加一个「当前工作状态」提示区，用 `flow.status`、`flow.current_node`、`flow.progress`、`flow.pending_reviews` 生成一句人话说明。
3. 根据状态计算按钮可用性：执行中禁用开始、继续、回退重跑；待确认/暂停/阻塞状态下引导用户先处理待确认事项，再继续。
4. 保留日志区作为技术细节，但页面主状态不再依赖用户读日志判断。
5. 未识别的状态或节点保留原文，避免错误翻译导致误导。

## 页面行为

- `ready`：显示「待开始」，提示用户点击开始执行流程。
- `running`：显示「执行中」，提示当前节点和进度；禁用继续与回退重跑。
- `review` / `blocked` / `paused`：显示「等待人工处理」或「已暂停」，提示待确认数量，并突出待确认入口。
- `done`：显示「已完成」，提示可查看质量摘要或输出归档。
- `failed`：显示「执行失败」，提示查看日志或错误信息。

## 按钮规则

- 开始：仅在非执行中状态可点，已完成时不作为主要操作。
- 暂停：仅执行中可点。
- 继续：执行中禁用；仅在待确认、暂停或阻塞状态可点。
- 回退重跑：执行中禁用；必须选择节点后才可点。
- 待确认：有待确认数量时显示更明确文案，例如「处理待确认（3）」。

## 测试与验证

- 运行前端构建，确认 TypeScript/Vite 构建通过。
- 手动检查流程执行页：不同状态下中文标签、提示语、按钮禁用规则符合预期。
- 若没有真实后端流程样本，至少通过现有 API 返回状态和静态页面检查验证不破坏页面渲染。
