# Q：规划表生成问题总结

> 记录于 2026-07-01。目的：说明 studio 现有规划表链路与《一课一练 规划表编写说明.md》的差距，以及它如何导致一课一练命名无法按权威版式完成。

## 一、结论速览

- studio 的规划表链路**整体没和《规划表编写说明.md》对齐**，属于“未实现/未对齐”，不是单点崩溃式 bug。
- 三个环节都有问题：**教材目录扫描（数据源）→ 规划生成/解析 → 数据模型**，且彼此**数据没接上**。
- 后果：一课一练**正文三行标题无法按权威版式生成**（尤其第 3 行的“第X单元…第Z章…第W节…”）。文件名基本能拼出来。

## 二、studio 现有规划链路（现状）

一课一练 flow 节点：`读取资料 → 解析目录 → 生成规划 → 知识点匹配 → 拉题与补题 → 质检导出 → 内容审阅 → 格式装配`
（`backend/engine/drivers/yikeyilian.py` + `backend/engine/runner.py`）。

1. **解析目录**（`kpoint_count` → `scan_textbook_toc`，`backend/shared/ocr/scanner.py`）
   - 只产出一个**纯文本 md**（`_教材目录扫描结果.md`）：优先读 PDF 内置书签，无书签则抽前 5 页文本片段。
   - **不结构化**、不产 `toc_structured.json`。

2. **生成规划**（`planning.gen_planning`，`backend/engine/steps/planning.py`）
   - 上传则 `_parse_uploaded`，否则 `_synthesize`（本地合成占位）。
   - 表头：`["序号", "卷号", "试卷主题", "考纲知识点", "卷型", "难度", "套数"]`。
   - `_parse_uploaded` 只提取 `试卷主题`(topic) 与 `考纲知识点`(point_name)，练号用行计数 `enumerate(...,1)`，忽略 A 列序号。

3. **落库**（`papers` 表，`backend/db.py`）
   - 字段：`id / project_id / paper_no / paper_type / module / topic / point_name / kpoint_id / status / docx_paths / qc_report_path`。
   - **无** 单元/章/节、级别、考纲标号、meta JSON 字段。

## 三、与权威规划表规范的差距（核心问题）

| # | 问题 | 现状 | 权威规范要求 |
|---|---|---|---|
| 1 | **数据源断裂** | `scan_textbook_toc` 只写 md，`planning.py` **完全不读**它 | 教材目录 → 结构化 → 驱动规划（§7.2） |
| 2 | **合成是占位** | `_synthesize` 造“课程 第i练主题”假数据，无层级 | 目录驱动，主题来自教材目录 |
| 3 | **解析不认层级** | 认第 1 行为表头、其余全当平铺数据行，用行号当练号 | 课程行/单元行(一级)/章行(二级) 合并行 + 考点行，练号取 A 列序号（§2.4/§2.6） |
| 4 | **列不一致** | 序号/卷号/试卷主题/考纲知识点/卷型/难度/套数 | A序号/B考纲知识点/C试卷主题/D级别/E题型/F难度/G套数/H考纲标号（§2.2） |
| 5 | **数据模型缺字段** | papers 只有 topic/point_name/module | 需 单元名/号、章名/号、节号、级别、考纲标号 |
| 6 | **级别逻辑缺失** | 无“极重要/重要/标准”，无极重要拆两行（一）（二） | §4.3/§5.1 硬性要求 |

## 四、接口层问题（有，且是关键卡点）

不是 HTTP 接口坏了，而是**内部数据契约（接口签名/行结构）没有承载层级的槽位**，即使解析出层级也传不下去：

1. `gen_planning(ctx, source) -> (Path, rows)`：`rows` 每条只有 `{paper_no, topic, point_name, paper_subtype, difficulty, status}`，**没有 unit/chapter/section/level 字段**。
2. `assemble(ctx, paper_no, qs)`（`backend/engine/steps/assemble.py`）取名只用 `topic`：
   `paper_name = paper.get("topic") or qs.meta.get("topic") or ctx.course`，再传给 `generate_docx(..., paper_name=paper_name, topic=paper_name)`。
3. `build_title_lines(ctx, paper_no, *, paper_name, paper_subtype, suffix, topic)`（`backend/shared/docx/naming.py`）：**签名里没有单元/章/节参数**，一课一练分支只用 province/textbook/edition/topic。
4. `scan_textbook_toc` 的产物**没有接口被 planning 调用**（数据流断点）。

## 五、一课一练命名能否完成？

分两层看：

- **文件名**：**能**。`name_template` 用到的字段都齐
  （`第{vol}练 {topic} {province}（{exam_type_name}）《{textbook}》（{edition}） 一课一练 （{variant}）`），与成品文件名规则基本一致。
- **正文三行标题**：**当前不能完整完成**。现在会产出一个三行标题（流程不报错、能出 docx），但和权威版式三处不符：
  - 行1 缺（考试类型）；
  - 行2 缺“第x练”且版次没括号；
  - **行3 完全没有“第X单元…第Z章…第W节…”层级**。
- **根因**：数据源（目录未结构化/未消费）→ 数据模型（无层级字段）→ 标题函数（无汉字序号转换、无三级拼接）三层全断。**要先把规划链路对齐，命名才可能完整完成。**

### 权威正文标题版式（目标）

```text
省份（考试类型）一课一练
《教材名》（出版社·版次） 第y练
第X单元 一级标题名称 第Z章 二级标题名称 第W节 试卷主题
```

- 单元/章/节用**中文大写序号**（第一单元/第一章/第一节）；练号 `第y练` 用 **A 列序号（阿拉伯）**。
- 一级标题=单元名，二级标题=章名，节名=C 列试卷主题。
- 参考旧工具包 `docx_generation.py` 的 `_number_to_cn / _strip_unit_title / _parse_section_title / _format_section_title`，含无 `unit` 时的“两级降级”分支。

示例：

```text
内蒙古自治区（对口招生）一课一练
《电子技术基础与技能》（高教版·第四版） 第1练
第一单元 二极管及其应用 第一章 二极管基础知识 第一节 二极管的结构与特性
```

## 六、要对齐需要改什么（实现清单）

1. `scan_textbook_toc` 输出**结构化目录**（单元/章/节层级 + 页码），或至少产 `toc_structured.json`。
2. 重写 `planning._parse_uploaded`：按 §2 层级行状态机解析（追踪当前单元/章），考点行取 A 列序号，记录 单元名/号、章名/号、节号、C 列主题、级别(D)、考纲标号(H)；未显式编号按出现顺序自动编号；极重要拆两行（一）（二）。
3. `_synthesize` / 目录驱动：让本地路径也能产层级（或明确要求上传标准 xlsx）。
4. **扩展数据契约**：`gen_planning` 的 row、`papers` 表（新增列或加 meta JSON）、`ctx`/`qs.meta`、`build_title_lines` 签名，全部补上层级字段。
5. 移植旧工具包 `_number_to_cn / _format_section_title`，重写 `build_title_lines` 一课一练分支为三行版式（单元/章/节汉字、练号阿拉伯），含两级降级分支。
6. 同步 `configs/yikeyilian/编写规范.md` §十一/§十二，并修掉“章节前缀/书名号”不一致。

## 七、前置决策（实现前需拍板）

- **规划表来源策略**：强制上传符合《规划表编写说明.md》的标准 xlsx？还是把本地教材目录扫描也做成结构化驱动（工作量更大）？
- OCR/合成路径在拿不到层级时的降级策略：走“两级标题模板”，还是直接要求补数据后再生成。

## 八、逐卷配置（多配置分区）——待规划表定型后实现

> 记录于 2026-07-01 的设计决策。目标：支持“不同卷号的试卷使用不同配置（题型题量/难度）”。
> 该特性依赖本文 §六 的规划表重构（尤其卷号语义），故排在其后实现。

### 8.1 需求模型（两层）

- **第一层 · 生成集 G**：沿用现有“卷号区间”，从 {1..总卷量} 选出要生成的子集（多区间、正整数、终止≥起始、互不重叠、不超总卷量）。
- **第二层 · 配置分区**：若干配置块，每块含自己的多区间 range，约束为：
  - 各块区间 ⊆ G；
  - 各块之间两两不重叠；
  - 各块并集 **恰好 == G**（每个待生成卷恰好归属一个配置块，无遗漏、无重复）。

### 8.2 数据契约

`projects.volume_config` 由“单份”升级为“分组”：

```json
{
  "paper_range": "2-3,5",
  "groups": [
    { "range": "2-3", "by_type": {…}, "difficulty": {…} },
    { "range": "5",   "by_type": {…}, "difficulty": {…} }
  ]
}
```

- 向后兼容：无 `groups` → 视为“单组覆盖 G 全部”，等价于现状。
- 消费点集中在 `build_paper_plan(ctx, paper_no)`（`backend/engine/steps/pull.py`）：按卷号命中所属分组，取该组 `by_type/difficulty`；命中不到时回退项目级/默认配置。

### 8.3 卷号语义（关键前置）——采用「保留原始卷号」

- **决策**：保留原始卷号（选 2、3、5 → 产出第 2、3、5 练；配置块按原始卷号匹配）。
- **前置改造**（依赖 §六）：现 `steps/planning.py` 会把所选子集重排成连续 1..N（见 §四/§一相关行），必须改为
  **持久化并沿用原始卷号**：
  1. `papers` 增加“原始卷号”槽位（新列或 meta），规划阶段写入所选原始卷号，不再 enumerate 重排；
  2. `build_paper_plan` 及配置分组匹配、文件命名“第 X 练”统一改用原始卷号；
  3. 影响面与 §六第 4 条“扩展数据契约”合并处理，避免二次返工。

### 8.4 校验规则

1. 生成集 G：多区间合法、不重叠、`max ≤ 总卷量`。
2. 配置分区：各块 ⊆ G、两两不重叠、并集 == G（缺卷/越界/重叠分别报错）。
3. 分值：非一课一练时，**每个配置块的 by_type 各自合计 = 满分**（现仅校验单份，需改为逐组校验）。

### 8.5 前端（创建向导第三步）

- 顶部保留“生成集”的区间编辑器；下方为可增删的“配置块列表”，每块内嵌多区间编辑器 + 该块题型题量/难度；
- 实时显示覆盖状态：已覆盖 G 的 X/共 Y 卷、未分配:[…]、重叠:[…]。

### 8.6 建议实现顺序

① 规划表保留/持久化原始卷号（并入 §六） → ② `build_paper_plan` 按原始卷号匹配 `groups` → ③ 第三步多配置块 UI + 逐组校验。

## 九、两套规划表结构与字段匹配对照

> 记录于 2026-07-02。样本与两份《规划表编写说明.md》严格一致，以下作为 studio 需对齐的目标 schema。

### 9.1 一课一练：单表（**扁平 11 列 A–K，2026-07-02 起启用**）

一门课程 = 一个 xlsx（多本教材拆多文件，序号各自从 1 起）。

> **格式变更（用户决策 2026-07-02）**：改为**扁平列式**——层级从"行承载（合并单元格）"改为"**列承载**"，第一列即课程，每行自带全列、**数据区无跨列合并**，便于后续变量读取。仅顶部标题/配置/教材 3 行做 `A:K` 合并居中（给人看）。

| 结构 | 内容 |
|------|------|
| 行1 | 标题 `{省份}{学科}《一课一练》考点规划表 v1`（合并 A1:K1） |
| 行2 | 配置 `题型：… \| 难度：…`（合并 A2:K2） |
| 行3 | 信息 `参考教材：《书名》…`（合并 A3:K3） |
| 行4 | 空行 |
| 行5 | 列标题（11 列 A–K） |
| 行6+ | 数据区：每行一个考点，自带 课程/单元/章，**无合并** |

11 列：**A 课程**、**B 单元**（一级标题）、**C 章**（二级标题）、**D 序号**（课程内连续，极重要占 2 个连续号）、**E 考纲知识点**（考纲原文、去尾句号）、**F 试卷主题**（≤10 字、去动词）、**G 级别**（极重要/重要/标准）、**H 题型**、**I 难度**、**J 套数=1**、**K 考纲标号**。三行标题的单元/章/节改从 B/C 列 + 序号推导（更稳）。

> **兼容**：解析器 `parse_8col` 双格式——自动识别新扁平表（表头含"课程/单元/章"）或旧合并表（一级行合并、二级行在 B 列、考点行 A=序号）；渲染 `render_8col` 一律输出新扁平格式。

### 9.2 考纲百套卷：三件套（同目录 `04_生成输出/生产规划/{省份} {考类}/`）

**(a) 考点规划总表 xlsx（10 列 A–J）**

| 列 | 含义 | 合并 |
|----|------|------|
| A 知识模块 | 课程/书名 | 同课程 |
| B 专题名称 | 考纲一级标题 ≤15 字 | 同专题 |
| C 考点名称 | AI 归纳、可作卷名 ≤15 字，覆盖 2–5 个 D | 否 |
| D 掌握/理解/了解考点内容 | 考纲细碎知识点（`1. 了解/理解/掌握…`，无尾句号） | 否 |
| E 考点训练卷数量=1 / F 考点训练卷卷号 | — | 否 |
| G/H 专题训练卷 数量=1/卷号 | — | 同专题 |
| I/J 课程综合卷 数量2–3/卷号区间 | — | 同课程 |

层级 **A⊃B⊃C⊃D**；末尾有合计行 + 每课程题型汇总表（100 分）。

**(b) 细目表 docx**：专题训练卷 + 课程综合卷各一份，逐题一行；字段含卷别/卷号/地区/课程/专题/题号/题型/难度/考查内容/对应考点(=C)/对应知识点(=D)/考纲要求/出题意图。**不填数字 ID**。

**(c) 映射表 xlsx（4 列）**：A 试卷序号、B 知识点 ID（学科网 kpointId，逗号分隔）、C 映射方式（`AI匹配`/`AI生成`/`聚合`）、D 备注。

### 9.3 字段匹配逻辑

- **一课一练**：教材目录(章/节/条目) → C 列主题；考纲原文匹配 → B 列知识点（本地关键词 + 可选 AI）；认知层次 → D 级别（掌握=极重要拆两行，理解/应用=重要，了解/其他=标准）；H 标号 `课程X§节(考点)`。
- **考纲百套卷**：考纲拆 → D 知识点，AI 归纳 → C 考点，一级标题 → B 专题，课程 → A；查《知识点题目数量.md》判窄考点(合计<80)按 5 条硬约束合并/减半；卷号 F(考点)→H(专题)→J(综合) 全局连续；映射表 D 文本 → 知识树 kpointId（**L3 叶子→L2 父，禁 L1，禁跨类**），考点卷 AI 匹配、专题/综合卷标 `聚合` 由代码合并去重。

### 9.4 与 studio 现状差距（含映射）

| 环节 | 权威规范 | studio 现状（文件） |
|------|----------|---------------------|
| 表头 | 一课一练 8 列 / 百套卷 10 列 | `planning.py` 仅 7 列 |
| 解析 | 层级状态机、A 列取序号、极重要拆行 | `_parse_uploaded` 平铺、`enumerate` 当卷号、重排丢原始卷号 |
| 数据源 | OCR 结构化 → 驱动 | `scan_textbook_toc` 只产 md，`planning.py` 不读 |
| 映射 | L3→L2 优先、禁 L1、窄考点合并、聚合 | `mapping.py` 仅本地树 + AI 兜底 + 阈值待确认，**无分层/无合并/无聚合** |
| 细目表 | 百套卷必产 docx | 无对应产出 |

## 十、规划表自动生成方案（LLM few-shot + 代码校验 + 人工确认）

> 记录于 2026-07-02。采用「LLM 出内容、代码出格式与校验、人工确认放行」的自动化方案，取代此前人工把样本+规范发给 AI 的手动流程。

### 10.1 分工

| LLM 产出（内容 + 编排 + 映射） | 代码只做（不生成内容） |
|-------------------------------|------------------------|
| 全部行内容 B/C/D、级别、极重要拆两行 | xlsx 样式渲染（合并/底纹/列宽/字体/边框） |
| A 列序号、卷号 F/H/J 编排 | docx 细目表渲染 |
| 窄考点合并与减半判断 | 规则校验器（见 10.3）+ 标红 |
| 映射 kpointId（L3/L2 选择、专题/综合卷聚合） | 人工确认闸门 + 回退重生成 |
| — | AI 调用留档（复用 `shared/ai/trace.py`） |

> 按用户决策：卷号编排与映射也交给 LLM。**准确性不靠限制 LLM 范围，而靠 10.3 校验器（硬拦截）+ 人工确认。**

### 10.2 两段式链路（读取 + 生成，各带一道可编辑确认闸门）

> 参照用户此前「Agent 全能体（读 OCR + LLM 生成）」的做法，落成**两段式**：读取归读取、生成归生成，中间产物可复核可改，避免一次不透明大调用错误叠加。

```text
考纲/教材 PDF
  ── 阶段1 抽取（按 PDF 类型分支）──────────────────────────
  · 文字版 PDF：PDF 有文本层 → PyMuPDF 抽文本层（scanner.py，免 OCR）
  · 图片版 PDF：无文本层 → fitz 渲染页图 → 视觉 OCR（vision.py，OCR 模型见 10.5）
  → 结构化抽取（教材目录层级 + 考纲条目）落盘 toc_structured.json
  → 【闸门1】界面简略展示，人工确认 / 编辑 / 增条目 ─★
  ── 阶段2 生成（LLM 出内容+卷号+映射）────────────────────
  → 组 prompt：规范要点 + 文本化样本(few-shot) + 已确认的抽取内容 (+ 知识树CSV)
  → call_api（shared/ai/llm.py）→ 返回 JSON 行
  → 渲染 8列/10列 xlsx + 细目表 docx（格式由代码保证）
  → 【闸门2】界面简略展示，人工确认 / 编辑 / 增条目 ─★
  → 校验器（见 10.3）→ 放行 / 回退重生成
```

### 10.3 校验器硬拦截项

列数=8 或 10、A 列序号连续、极重要必须成对、卷号全局无跳号/无重复、B/D 去尾句号、C 列字数上限、映射**禁 L1/禁跨类**、映射 A 列覆盖全部试卷、专题/综合卷聚合 ID = 其下考点卷并集去重。语义类判断（如"内容相近")代码无法确定性判定时，标注「仅人工确认可放行」。

### 10.4 说明

- 不强制上传标准 xlsx：既可解析上传的标准 xlsx，也可（文字版/图片版）PDF 目录驱动合成。
- 样本 + 规范用于 LLM 侧的 few-shot（样本需先转成文本表，二进制 xlsx 不能直喂）。
- 生成后进入「生成 + 人工确认」闭环，不达标行标红回退。
- **两道闸门均为界面内可编辑**（简略展示即可，支持增条目），复用 review/待确认机制；「可编辑」是新增前端能力（现「中间文件」页只读）。

### 10.5 模型与 PDF 类型

- **主 LLM model 取自配置** `get_llm_config().model`（settings `llm.model` > `LLM_MODEL` > 兜底 `gpt-4o`）；当前实际配置为 **gpt-5.5**。源码里的 `gpt-4o` 仅为未配置时的兜底默认，非实际使用模型。
- **OCR/视觉走独立配置** `get_vision_config()`（`vision.model` 等），与主 LLM 分离；**具体 OCR 模型待定、可插拔**（gpt-5.5 是否支持读图未确认，图片版 PDF 可另配专用 OCR 模型）。`vision.py` 现为空壳，需落地实现。
- PDF 类型判定：逐页取文本层，前若干页文字量低于阈值 → 判为图片版走视觉 OCR，否则走文本层抽取。
