# 自动化修正平台实现步骤

## 1. 目标定位

在现有“结构化人工审核队列”的基础上，进一步扩展成一个“自动化修正平台”。平台目标不是只记录问题，而是形成完整闭环：

1. 从质检报告、API 拉题日志、人工反馈中收集错误。
2. 将每个错误转成结构化修正任务。
3. 判断任务适合自动修正还是人工修正。
4. 对可自动处理的问题调用 API / 模型生成修正内容。
5. 将修正结果写回结构化题目数据或 Word 文档。
6. 重新生成 / 覆盖目标 DOCX。
7. 留存修正前后对比、处理状态和复核记录。

最终效果：

- 人工不再翻散乱日志。
- 简单错误可自动修。
- 复杂错误可进入人工界面集中处理。
- 每一处改动都有来源、动作、结果和回溯记录。

---

## 2. 平台总体流程

```text
质检 / API 拉题 / 人工反馈
        ↓
错误采集器
        ↓
review_queue.json 修正队列
        ↓
任务分流
  ├── 自动修正任务 → 调用 API / 模型 → 生成修正内容
  └── 人工修正任务 → 人工确认 / 编辑
        ↓
结构化题目更新
        ↓
重新生成或修正 Word
        ↓
二次质检
        ↓
修正记录落盘
```

## 3. 核心数据结构

### 3.1 人工 / 自动审核队列：`review_queue.json`

建议放置路径：

```text
04_生成输出/人工审核/review_queue.json
```

每条记录建议包含：

```json
{
  "id": "review-000001",
  "paper_label": "第12卷",
  "paper_type": "考点训练卷",
  "source_docx": "04_生成输出/组卷待质检/重庆市 电子信息类/第12卷 xxx.docx",
  "question_no": 5,
  "question_type": "单选题",
  "issue_type": "answer_missing",
  "severity": "error",
  "description": "第5题缺少答案",
  "suggested_action": "补齐答案并重新生成解析",
  "fix_mode": "auto",
  "status": "pending",
  "source_report": "04_生成输出/质检报告/xxx.json",
  "created_at": "2026-06-25T00:00:00",
  "updated_at": "2026-06-25T00:00:00",
  "before": {
    "stem": "...",
    "answer": ""
  },
  "after": null,
  "fix_result_path": null,
  "notes": ""
}
```

### 3.2 字段说明

| 字段 | 说明 |
|---|---|
| `id` | 审核任务唯一 ID |
| `paper_label` | 卷号，如第12卷 |
| `source_docx` | 原始 Word 路径 |
| `question_no` | 问题题号 |
| `issue_type` | 错误类型，如答案缺失、解析缺失、题干过短、重复题 |
| `severity` | 严重程度：`error / warning / info` |
| `fix_mode` | `auto / manual / hybrid` |
| `status` | `pending / fixed / ignored / need_review / failed` |
| `before` | 修正前内容快照 |
| `after` | 修正后内容快照 |
| `fix_result_path` | 修正结果文件路径 |

## 4. 错误类型分级

### 4.1 适合自动修正的问题

这些问题可以优先调用 API / 模型自动处理：

- 答案缺失
- 解析缺失
- 判断题答案格式不统一
- 选项格式轻微错误
- 题干中存在明显多余空格、乱码、HTML 残留
- 分值格式不规范
- 题型名称不统一
- 缺少题号或题号不连续
- AI 生成题中缺少解析或答案字段

### 4.2 适合人工确认后自动修正的问题

这些问题建议设为 `hybrid`：

- 题干疑似表述不完整
- 答案与解析不一致
- 题目难度不符合规划
- 题目疑似跨知识点
- 多题重复或高度相似
- API 拉题内容和当前考点匹配度不足

流程是：

1. 系统给出建议修正方案。
2. 人工确认是否采用。
3. 确认后系统写回题目或 Word。

### 4.3 必须人工处理的问题

这些问题不建议直接自动改：

- 题目专业知识明显错误，但无法自动判断正确答案
- 图片、公式、表格缺失或错位
- 题目来源有版权或引用疑问
- 需要重新选题或整题替换
- 需要人工决定是否删除该题

## 5. 实现步骤

## 阶段一：结构化审核队列

### 5.1 建立队列输出目录

新增目录：

```text
04_生成输出/人工审核/
```

新增文件：

```text
04_生成输出/人工审核/review_queue.json
04_生成输出/人工审核/fix_results/
04_生成输出/人工审核/fix_logs/
```

### 5.2 从质检报告收集错误

在质检完成后，读取已有报告：

```text
04_生成输出/质检报告/*.json
04_生成输出/质检报告/*.md
```

将其中需要人工处理或自动修复的问题转成 `review_queue.json` 记录。

### 5.3 从 API 拉题日志收集错误

结合 P0 第 5 项新增的 API 字段链路日志，将以下问题写入队列：

- API 返回字段缺失
- `kpointIds` 为空
- `difficulty` 缺失
- `typeId` 与目标题型不匹配
- 返回题量不足
- API 返回结构异常

### 5.4 去重规则

同一问题不要重复写入。建议去重键：

```text
paper_label + question_no + issue_type + source_report
```

如果同一问题已存在，则只更新 `updated_at` 和最新说明。

## 阶段二：自动修正引擎

### 5.5 新增修正调度器

建议新增模块：

```text
01_工具脚本/生成器/review_queue.py
01_工具脚本/生成器/auto_fix_engine.py
```

职责：

- `review_queue.py`
  - 读写 `review_queue.json`
  - 新增任务
  - 更新状态
  - 查询待处理任务

- `auto_fix_engine.py`
  - 判断任务是否可自动修正
  - 构造修正 prompt
  - 调用 API / 模型
  - 校验修正结果
  - 保存修正前后对比

### 5.6 自动修正 API 调用方式

复用现有配置读取和 API 调用能力：

```text
01_工具脚本/生成器/config_io.py
```

优先复用已有：

- `load_config()`
- `call_api()`
- token 用量统计逻辑

避免重新写一套模型调用。

### 5.7 自动修正 Prompt 输入

每次自动修正至少传入：

- 卷号
- 题号
- 题型
- 原题干
- 原选项
- 原答案
- 原解析
- 质检问题
- 当前规划约束
- 允许修改范围

示例：

```text
请修正以下题目的“答案缺失”问题。
只允许补充 answer 和 analysis，不要改题干和选项。
返回 JSON：{"answer": "...", "analysis": "..."}
```

### 5.8 自动修正结果校验

自动修正后不能直接写回，必须做基础校验：

- 返回必须是 JSON
- 必填字段不为空
- 题型与答案格式匹配
- 判断题答案统一为 √ / ×
- 单选题答案必须是 A-D
- 多选题答案必须是 A-D 的组合
- 解析不能过短

校验失败则：

```json
"status": "need_review",
"fix_mode": "manual"
```

## 阶段三：Word / 结构化题目写回

### 5.9 优先修改结构化题目 JSON

不要直接把 Word 当作唯一数据源。推荐流程是：

1. 从 Word 拆题，得到结构化题目 JSON。
2. 对 JSON 中对应题目做修正。
3. 再调用现有 DOCX 生成流程重新生成 Word。

这样比直接改 DOCX 更稳定。

可复用现有模块：

```text
01_工具脚本/生成器/paper_loader.py
01_工具脚本/生成器/paper_assembler.py
01_工具脚本/生成器/docx_generation.py
01_工具脚本/生成器/runner.py
```

### 5.10 直接修正 Word 的适用场景

只有在以下问题中考虑直接改 DOCX：

- 页眉页脚
- 标题文字
- 简单格式错误
- 题号显示错误
- 明显文字替换

复杂题目内容仍建议走“结构化题目 → 重新生成 DOCX”。

### 5.11 写回策略

每次修正都必须保留原文件：

```text
原文件：第12卷 xxx.docx
备份：04_生成输出/人工审核/backups/第12卷 xxx.before_fix.docx
修正后：第12卷 xxx.fixed.docx
```

确认无误后，再由人工或命令选择是否覆盖原文件。

## 阶段四：人工修正界面 / 命令

### 5.12 最小可用命令

可以先不做图形界面，先提供命令行：

```bash
python create.py --review-status
python create.py --review-list
python create.py --review-fix-auto
python create.py --review-export
python create.py --review-apply review-000001
python create.py --review-ignore review-000001
```

### 5.13 命令说明

| 命令 | 作用 |
|---|---|
| `--review-status` | 查看待审核统计 |
| `--review-list` | 列出待处理任务 |
| `--review-fix-auto` | 批量自动修正可自动处理的问题 |
| `--review-export` | 导出人工审核 Markdown / Excel |
| `--review-apply <id>` | 应用某条修正结果 |
| `--review-ignore <id>` | 标记忽略 |

### 5.14 人工审核导出格式

导出 Markdown 示例：

```markdown
# 人工审核队列

- 问题：答案缺失
- 建议动作：补齐答案和解析
- 原题路径：...
- 质检报告：...
- 当前状态：pending

### 原题
...

### 系统建议修正
...

### 人工处理意见
- [ ] 采用系统建议
- [ ] 手动修改
- [ ] 忽略
```

## 阶段五：二次质检和闭环

### 5.15 修正后自动复检

每次应用修正后，自动执行相关检查：

- 单题规则检查
- 整卷题量检查
- 答案格式检查
- 解析完整性检查
- 跨卷重复检查（必要时）

如果复检通过：

```json
"status": "fixed"
```

如果复检失败：

```json
"status": "need_review"
```

### 5.16 修正结果落盘

每条任务生成一个修正结果文件：

```text
04_生成输出/人工审核/fix_results/review-000001.json
```

内容包括：

```json
{
  "review_id": "review-000001",
  "paper_label": "第12卷",
  "question_no": 5,
  "issue_type": "answer_missing",
  "fix_mode": "auto",
  "status": "fixed",
  "before": {},
  "after": {},
  "api_prompt": "...",
  "api_response": {},
  "validation": {
    "passed": true,
    "issues": []
  }
}
```

## 6. 推荐开发顺序

### 第一步：只做队列，不自动改

目标：先把问题集中起来。

- 新建 `review_queue.json`
- 从质检报告收集问题
- 支持查看待处理任务
- 支持标记 `ignored / done`

### 第二步：做自动修正建议，不写回

目标：让系统生成“建议修正”。

- 调用 API / 模型
- 生成 `after` 建议
- 保存修正前后对比
- 人工确认是否采用

### 第三步：写回结构化题目 JSON

目标：先改结构化数据，不直接改 Word。

- 定位对应卷、对应题
- 写回题目 JSON
- 重新生成 DOCX
- 保存修正版本

### 第四步：自动复检

目标：确认修正真的有效。

- 自动跑质检
- 更新任务状态
- 失败则回到人工审核

### 第五步：扩展到 Word 级修正

目标：处理少量格式类问题。

- 页眉页脚
- 标题
- 题号
- 简单文本替换

## 7. 最小可行版本 MVP

MVP 不需要一次做完整平台，只需做到：

1. 生成 `review_queue.json`
2. 支持列出待处理问题
3. 支持对答案/解析缺失调用 API 自动生成建议
4. 保存修正前后对比
5. 人工确认后写回结构化题目
6. 重新生成 DOCX
7. 二次质检并更新状态

MVP 完成后，就能覆盖最常见的“机器发现错误 → 自动建议 → 人工确认 → 自动重出 Word”流程。

---

## 8. 风险与注意事项

1. **不要直接覆盖原 Word**  
   必须先备份，再生成修正版。

2. **自动修正不能绕过质检**  
   API 生成内容必须经过规则检查。

3. **复杂专业判断必须留给人工**  
   不要让模型直接决定专业正确性。

4. **保留修正前后对比**  
   方便回滚和追责。

5. **队列状态必须可恢复**  
   中途失败后，下次运行应能继续处理未完成任务。

## 9. 与当前 P1 / P2 清单的关系

这个平台可以看作是 P1 第 9 项“结构化人工审核队列”的增强版：

- 原第 9 项：集中记录人工审核问题。
- 扩展后：集中记录 + 自动修正建议 + 人工确认 + 写回 Word + 二次质检。

它也会覆盖 P1 第 10 项的一部分目标：

- API 缺题信息落盘
- 修复结果落盘
- 修复前后对比落盘
- 单卷问题可回溯

因此建议开发时把两项合并设计，但分阶段实现。
