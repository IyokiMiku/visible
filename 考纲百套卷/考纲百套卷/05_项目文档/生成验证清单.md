# 生成验证清单

## 一、静态检查

- [ ] `runner.py` 能正常编译
  - `python -m py_compile 01_工具脚本/生成器/runner.py`
  - `python -m py_compile 01_工具脚本/生成器/_topic_pool.py`

- [ ] 搜索关键规则是否已落地
  - `hit_rate` 只在受控位置使用
  - `topic_kpoint_ids` 不再全量合并映射表
  - `_select_by_source_dedup` 不再突破每组 3 道
  - 无 DOCX 时默认先 API 拉题，不直接 AI 整卷

## 二、单卷验证：已有原文档

选一套 `04_生成输出/组卷待质检` 中已有原 DOCX 的卷。

### 情况 A：题量足够
- [ ] 运行生成流程
- [ ] 结果应直接进入质检 / 修复 / DOCX 生成
- [ ] 不应触发 API 拉题
- [ ] 不应触发 AI 补题

### 情况 B：题量不足
- [ ] 运行生成流程
- [ ] 应先识别出“题量不足”
- [ ] 应按规划表补题
- [ ] 若题量超过规划，应自动裁剪到规划数
- [ ] 若补题后仍低于 40%，应只补到规划一半，且提示每题分值 ×2
- [ ] 若补题后达到 / 超过 40%，应补齐到规划数

### 需要重点确认的输出
- [ ] 保存的拆题 JSON 题数与规划一致或符合半量规则
- [ ] 质检报告正常生成
- [ ] 生成的 DOCX 文件数量正确
- [ ] `错误收集.md` 只在有待人工审核问题时追加

## 三、单卷验证：没有原文档

选一套 `组卷待质检` 中没有 DOCX 的卷。

- [ ] 运行生成流程
- [ ] 应先走 API 拉题
- [ ] 成功后生成待质检 DOCX
- [ ] 再进入质检 / 修复 / DOCX 生成
- [ ] 不应一开始就直接 AI 整卷

如果 API 拉题失败：

- [ ] 才允许进入 AI 兜底
- [ ] AI 行为应有明确提示，不应静默跳过

## 四、题库 / 专题池验证

选同一专题下的两三套卷验证。

- [ ] 第一套卷使用后，题目被标记为已用
- [ ] 后续同专题卷不再重复使用这些题
- [ ] 同专题卷共用小题库文件
- [ ] 小题库文件在专题处理结束后被清理

重点看：

- [ ] `used_by` 是否写成当前卷号
- [ ] 同题不再重复出现在后续卷中

## 五、来源分组上限验证

准备或模拟同 `year + source / 试卷名` 的候选题超过 3 道。

- [ ] 同组最多只选 3 道
- [ ] fallback 不应突破上限
- [ ] 不同组仍可继续补足

## 六、40% 分界验证

准备两种样例：

### 样例 1：已选题数 < 40%
- [ ] 最终只补到规划题量一半
- [ ] DOCX 提示每题分值 ×2
- [ ] 总题数不超过半量目标

### 样例 2：已选题数 ≥ 40%
- [ ] AI 补到完整规划题量
- [ ] 不应提示分值翻倍
- [ ] 最终题数等于规划数

## 七、过量题目验证

- [ ] 当原文档或拉题结果超出规划题量时
- [ ] 应随机裁剪到规划数
- [ ] 裁剪后题型分布仍应尽量符合规划

## 八、最终交付验证

- [ ] 原卷版 DOCX 生成成功
- [ ] 解析版 DOCX 生成成功
- [ ] `zip` 或后处理输出正常
- [ ] `auto_review` 报告无致命错误
- [ ] 质检状态与预期一致

如需按“考点卷 / 专题卷 / 综合卷”拆分，可继续细化为分组版清单。
